TL;DR: The fundamental rule of state placement: state should live as close to where it’s used as possible. Local state (useState in one component) is the default. Elevate to shared/global state only when: (1) multiple components in different subtrees need the same value, OR (2) the state needs to survive component unmounting. The anti-pattern: making everything global “just in case” (creates unnecessary coupling and reduces the ability to reuse components). The over-correction: keeping everything local until you have deep prop drilling chains (creates unnecessary complexity and fragility). The right heuristic: “what is the smallest subtree that contains all components that need this state?” - that subtree’s root is where the state should live.

🔥 The Problem This Solves

A team is building a multi-step checkout wizard: Step 1 (shipping address), Step 2 (payment), Step 3 (review). The form data from Step 1 needs to be shown in Step 3. Developer A puts all form state in a global Redux store (accessible everywhere). Developer B passes it as props from a parent Checkout component.

Developer A’s choice: global state for form data that only the checkout flow needs. Now: any component anywhere can access or modify checkout form data. Accidental modification from unrelated components is possible. The Checkout component can’t be used standalone (depends on the global store being configured). Unit testing requires a full store setup.

Developer B’s choice: local state in the Checkout parent, passed as props to steps. The state is isolated to the checkout subtree. The Checkout component is self-contained and testable. No unrelated components can accidentally modify it.

The lesson: not all shared state should be global. “Shared between checkout steps” = local to the Checkout parent. “Shared across the whole app” (e.g., authenticated user) = global.

📘 Textbook Definition

Local state: State managed by a single component with useState or useReducer. Only that component and its descendants (via props) can read or modify the state. When the component unmounts: the state is lost.

Lifted state: State elevated from one or more child components to their common ancestor. The ancestor holds the state and passes it down as props. Appropriate when: multiple sibling components need access to the same state.

Global state: State accessible by any component in the application tree, typically via a global store (Zustand, Redux) or React Context. Appropriate for: state needed by many components in unrelated parts of the tree, or state that must persist across component mount/unmount cycles.

Colocation: The practice of keeping code (including state) as close as possible to where it is used. Applied to state: a state variable that’s only used in one component should be in that component, not in a parent or global store.

State ownership: The component or store that holds the authoritative copy of a piece of state. Ownership determines: who can modify the state directly (the owner), and who receives the state as a prop or via subscription (all other consumers).

🔩 First Principles Explanation

Why local state is the default: Local state has the smallest blast radius. A bug in how local state is managed: affects one component. A bug in global state management: can affect any component in the app. Local state: easier to test (no store setup needed), easier to reason about (all state transitions are in the component file), and easier to reuse (components that own their state can be dropped into any app).

Why lifting state is the right intermediate step: Before making state global, ask: do these components have a common ancestor? If yes: lift the state to that ancestor. This keeps the state as local as possible while sharing it. The ancestor becomes the “owner” - it holds the authoritative value and passes it down as props or provides a setter function for children to call. Global state stores should be the last resort, not the first choice.

Why global state is overused: Global state is easy to add (no prop threading required) but has hidden costs: (1) any component can accidentally read or modify any global state, (2) components become coupled to the global store structure (changing the store shape breaks all consumers), (3) testing requires a full store setup, (4) understanding a component requires understanding the global state, not just the component file. The ease of adding global state is its greatest danger: developers reach for it to avoid the “prop threading” cost, without considering the hidden long-term costs.

THE TRADE-OFFS:

Local vs global for component-specific state: Local: isolated, testable, no coupling. Global: accessible anywhere, accessible to tools (devtools). For UI state like “is dropdown open”: local is always better. For cart items: global is better (accessed from many places). The heuristic: if you wouldn’t feel uncomfortable exposing this state to any component in the app, it can be global.

Lifting state vs prop drilling: Lifting state is correct when the common ancestor is close (1-2 levels up). When the common ancestor is 5+ levels up: prop drilling all the way becomes painful (each intermediate component must pass props it doesn’t use). At that point: Context or global state becomes justified.

ESSENTIAL vs ACCIDENTAL COMPLEXITY:

Essential: Sharing cart item count between the cart button in the header and the cart page.

Accidental: Putting “is the image carousel on slide 3” in the global Redux store because it’s slightly easier than managing it locally.

🧪 Thought Experiment

A music player component (<MusicPlayer>) has these state variables: currentTrack (the playing track), currentTime (playback position in seconds), volume (0-100), isPlaying (boolean), playlistId (the playlist being played), isPlayerMinimized (the UI size state).

Which state variables should be local, and which should be global?

Global candidates:

  • currentTrack: the nav bar shows the track name. The queue sidebar shows the track. Multiple components need it -> global.
  • isPlaying: affects the play button everywhere, notifications -> global.
  • playlistId: the playlist page highlights the active playlist -> global.

Local candidates:

  • currentTime: only the progress bar inside the player shows it. No other component needs it (they don’t show seconds). Updating it 60fps in global state would cause unnecessary re-renders everywhere -> local.
  • volume: if other components need to show/change volume: global. If volume is only in the player widget: local.
  • isPlayerMinimized: purely visual, only the player component needs it -> local.

The exercise: not all state in one component belongs in the same place. Some is globally shared, some is truly local.

🧠 Mental Model / Analogy

Think of state as physical objects in an office building. A sticky note on your monitor (local state): only you can see or move it. Perfect for things only you need to remember. Notes on the team whiteboard (lifted state): your immediate team can see it - the right scope for things your team shares. The company wiki (global state): anyone in the company can read it, anyone with access can modify it. The company wiki is valuable for company-wide information (current product roadmap) but terrible for personal reminders (your sticky note reminding you to reply to a specific email). The failure mode: putting everything on the company wiki. Now: nobody knows who owns what, anyone can delete anything, and finding your personal reminder requires searching the whole wiki. State management follows the same principle: global state is the company wiki - valuable for company-wide facts, inappropriate for team or personal information. Defaulting to global: the organizational equivalent of everyone posting their sticky notes to the company wiki.

📶 Gradual Depth - Five Levels

Level 1 - Student: Keep state in the component that uses it. If two components both need it, put it in their common parent and pass it down as props. Only make state global when many unrelated components need it.

Level 2 - Junior Dev: Apply the “smallest subtree” rule: draw the component tree, find all components that need the state, find their lowest common ancestor - that’s where the state lives. If the LCA is too far up the tree (causing prop drilling): use Context or a global store.

Level 3 - Mid Dev: Distinguish between “global because multiple components need it” and “global because it’s the lazy choice.” Audit global state: is each piece genuinely needed globally, or can it be localized? Consider “lifting state” as an intermediate step before globalizing.

Level 4 - Senior Dev: Performance implications: local state updates trigger re-renders only in the component tree below the state holder. Global state updates in Context trigger re-renders in ALL consumers. Zustand/Redux with selectors: only re-renders components whose selected slice changed. Design global state with performance-aware selectors.

Level 5 - Staff/Principal: State ownership is a team concern, not just a component concern. Which team owns which global state? How do you prevent cross-team state mutation? API boundaries between teams in a micro-frontend architecture: state that crosses micro-frontend boundaries must be globally shared, but only at the boundary level. Each micro- frontend owns its internal state; global state is only what’s necessary for cross-boundary coordination.

⚙️ How It Works (Mechanism)

State placement decision algorithm:

1. Identify consumers: which components need this state?
   List them all.

2. Find lowest common ancestor (LCA) in the tree:
   [ProductPage]
     [ProductDetails]       <-- uses state
     [Cart]
       [CartItem]           <-- uses state
   LCA = [ProductPage]
   Decision: lift to ProductPage

3. Is the LCA reasonable?
   1-3 levels above closest consumer: YES -> lift
   4+ levels above closest consumer: NO -> global

4. Is it UI state (dropdown, tooltip)?
   YES -> always local, never global
   NO -> continue to step 1

5. Does it need to survive route changes?
   YES -> global (or URL state)
   NO -> lift to LCA or keep local

🔄 The Complete Picture - End-to-End Flow

State elevation lifecycle:

Start: single component uses state (local)

Feature grows: sibling component needs same state
  -> Lift to common parent

Feature grows more: distant cousin component needs it
  -> Option A: lift further (prop drilling)
  -> Option B: Context (if stable value)
  -> Option C: Global store (if frequent updates)

Signs you've over-globalized:
  Global store has UI-specific flags
  Unit tests require full store setup for simple
  components
  Removing a component requires global store cleanup

Signs you've under-globalized:
  Prop chains pass same value through 5+ components
  Duplicated state in sibling components
  "Out of sync" bugs between siblings

  return (
    <img
      src={src}
      alt={alt}
      onClick={() => setIsZoomed(!isZoomed)}
      className={isZoomed ? 'zoomed' : ''}
    />
  );
  // No global store needed. Zoom state is local.
  // Testing: no store setup, just render and click.
}

// GOOD: Lift state to common ancestor
function ProductPage() {
  // Shared between ProductDetails and CartButton
  // LCA is ProductPage (1 level above both)
  const [quantity, setQuantity] = useState(1);

  return (
    <div>
      <QuantitySelector
        value={quantity}
        onChange={setQuantity}
      />
      <AddToCartButton
        quantity={quantity}  // uses lifted state
      />
    </div>
  );
}

// GOOD: Global state for truly global data
const useCartStore = create(set => ({
  items: [],  // needed across header, cart page, checkout
  addItem: (item) => set(s => ({
    items: [...s.items, item]
  })),
  removeItem: (id) => set(s => ({
    items: s.items.filter(i => i.id !== id)
  })),

Code walkthrough: Three patterns. (1) BAD global state: UI-only state (dropdown open, image zoomed, tooltip visible, carousel slide) in a global Zustand store. These should never be global: they’re component- specific, ephemeral, and coupling them to the global store means any component can accidentally toggle another component’s dropdown. (2) BAD prop drilling: onCartUpdate passes through Header and Nav even though neither uses it. If the cart callback signature changes: three files change. (3) GOOD colocated UI state: isZoomed lives in ProductImage. Only ProductImage needs to know if it’s zoomed. Testing is trivial (no store). (4) GOOD lifted state: quantity is used by both QuantitySelector and AddToCartButton. Their LCA is ProductPage (1 level up). Lift to ProductPage, pass down as props. (5) GOOD global state: cartItems is genuinely needed across the header (cart count badge), the cart page, and the checkout. Global is correct here.

⚠️ Common Misconceptions

Misconception 1: “Global state is easier, so default to it.”

Reality: Global state is easier to ADD but harder to MAINTAIN. The setup cost is one line (add to store), but the long-term costs accumulate: any component can accidentally read/write any global state, removing a feature requires global store cleanup, unit testing requires full store setup, and refactoring the store shape breaks all consumers. Local state costs more upfront (pass as props) but has zero long-term coupling costs.

Misconception 2: “Component state doesn’t scale for large applications.”

Reality: Large applications can have most state as local or lifted state, with only a small amount truly global. The notion that “you need a global state manager for large apps” is a legacy from before modern React patterns. Server state (React Query), URL state (router), and form state (React Hook Form) handle most state in a “large app.” What’s left for a global store is surprisingly small: auth state, user preferences, and a few UI settings. The scale of an application doesn’t dictate global state; the number of genuinely-global state concerns does.

Misconception 3: “Lifting state is only for component props.”

Reality: “Lifting state” is the pattern of moving state to a common ancestor. This applies to React components (lifting to a parent), but the same principle applies across frameworks and systems: in Redux, “lifting” happens to the store level; in server-side rendering, state is “lifted” to the server and passed down via props; in micro-frontends, state is “lifted” to a shared event bus or shared global. The principle (state lives at the lowest level that needs to share it) is universal.

🚨 Failure Modes & Diagnosis

Failure 1: Global state for UI-only state (testing difficulty and hidden coupling)

  • Cause: Developer put modal/tooltip/dropdown state in global store to avoid passing props. Works initially; causes problems in testing and code reuse
  • Diagnosis: Check global store for state variables named is*Open, is*Visible, active*Tab, current*Slide. If found: likely UI state that should be local
  • Fix: Move to component local state. Add a <DropdownContainer> component that owns its open state if needed across siblings

Failure 2: Prop drilling (state passed through 3+ intermediate components that don’t use it)

  • Cause: State was lifted too high, or the component tree grew after the initial design
  • Diagnosis: Find props that are passed through components without being used (just { onX, valueY, ...rest } patterns)
  • Fix: If components are close cousins: Context. If they’re distantly related: global store. If they’re related but not close: refactor component composition (use children/render props)

💎 Transferable Wisdom

The local-vs-global state mental model is the application of “principle of least privilege” to data access: each piece of code should only have access to the state it needs to do its job. This is a security and stability principle. Applied to state: components should only access the state they need; other components should not be able to access or accidentally modify it. This principle appears in many forms: microservices (each service owns its data, other services request via API rather than direct database access), filesystem permissions (each process has the minimum permissions needed), database access control (each application user has only the columns they need). The colocate-state principle is “principle of least privilege” for UI state. Applying it consistently: reduces the blast radius of bugs, simplifies testing, and makes components reusable across applications.

💡 The Surprising Truth

React’s founder Jordan Walke said in a 2015 interview that one of the original design goals of React was to make state management “obvious” by forcing components to declare what state they need (props) rather than reaching into shared memory. The design pushed toward component-local state and explicit prop passing. Yet within a year: the community had built Redux (global state) as the standard React pattern. The surprising implication: the framework was designed to discourage global state, but the community adopted global state as the default. This created a generation of React developers who reach for global state first. The recent “state colocation movement” (return to local state, React Query for server state, URL state for navigation state) is actually returning to React’s original design intent. “Lifting state” was always the intended pattern; global stores were always a workaround for cases where lifting wasn’t practical. Most developers have these inverted: they reach for global stores first and use local state as the workaround.

🎯 Interview Deep-Dive

Q: What is your approach to deciding between local and global state?

A: I start with the smallest scope and escalate only when necessary.

The default: local state in the component that uses it. useState, no external dependencies, easiest to test, no coupling. When a component unmounts: state is garbage collected automatically.

The first escalation: if two sibling components need the same state, I lift to their common ancestor. The ancestor holds the state and passes it as props. This keeps the state as local as possible while sharing it. The “smallest subtree” rule: state lives in the root of the smallest subtree that contains all its consumers.

The second escalation: when prop drilling becomes painful (state passing through 3+ components that don’t use it, just forwarding it), or when the common ancestor is too far up the tree. At this point: if the value changes rarely (auth user, theme): React Context. If it changes frequently (cart items, filters that update on every selection): Zustand or another global store with selector-based subscriptions (to avoid re-rendering non-subscribed consumers).

What I explicitly keep local: UI state (is this dropdown open, is this tooltip visible, is this image zoomed). These should never be in a global store. They’re component-specific, ephemeral, and coupling them to a global store makes the component impossible to test or reuse without the store being set up.

The warning sign I look for in code reviews: global state with names like isHeaderDropdownOpen, activeCarouselSlide, or isTooltipVisible. These are UI state masquerading as global state. I move them to local state and remove the global coupling.
