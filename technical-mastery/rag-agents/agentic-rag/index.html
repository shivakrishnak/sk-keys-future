TL;DR: Agentic RAG replaces the fixed pipeline (embed -> search -> generate) with an LLM that DECIDES whether to retrieve, WHICH knowledge source to query, and WHETHER the retrieved context is sufficient or requires follow-up retrieval. The retrieval step becomes a TOOL the agent can invoke zero or more times, not a fixed pipeline stage. In practice, most “agentic RAG” systems are implemented as LangGraph StateGraphs with conditional routing: a routing node classifies the query and routes it to the appropriate retrieval source or to a direct-answer path. True iterative agentic RAG (the agent retrieves, evaluates, retrieves again if needed, in a loop) is rare in production because each retrieval loop adds 500-1,500ms and an LLM call ($0.003-$0.015). The architectural shift is significant: retrieval quality depends on LLM reasoning quality, not just the embedding + similarity score. This creates a new failure mode (the LLM decides not to retrieve when it should) that is invisible in standard RAGAS metrics and requires agent-specific evaluation.

🔥 The Problem This Solves

Standard RAG has a structural limitation: it ALWAYS retrieves, it retrieves from ONE source, and it retrieves ONCE. This architecture fails when:

  1. The query does not need retrieval: “What is 2+2?” triggers a vector search in a standard RAG pipeline, wasting latency and cost on an unnecessary retrieval step.
  2. Multiple knowledge sources exist: A system with a product FAQ, a policies database, and a ticketing system cannot route each query to the correct source using a single vector search.
  3. One retrieval pass is insufficient: “Compare the pricing policies for product A and product B” requires two retrieval calls (one per product), then synthesis. A single retrieval pass returns mixed context that confuses the LLM.
  4. The answer requires tool use + retrieval: “What is our current ARR and how does it compare to last quarter?” requires a database query (ARR) AND a document retrieval (industry comparison). Standard RAG cannot call a database.

Agentic RAG solves all four by giving the LLM the ability to decide retrieval strategy dynamically.

📘 Textbook Definition

Agentic RAG is an architecture in which an LLM acts as an agent that DECIDES when to retrieve, WHAT to retrieve from, and whether the retrieved context is SUFFICIENT to answer a query. Retrieval is implemented as a TOOL the agent can call, rather than a fixed first step in a pipeline.

Key variants by complexity:

  • Query Router (simplest): Classify query type -> route to correct retrieval source. One LLM call for classification; no iterative retrieval.
  • Self-RAG: LLM generates retrieval tokens (“Retrieve”, “No Retrieve”) interleaved with generation; model decides mid-generation whether to retrieve.
  • Corrective RAG: After retrieval, evaluate context relevance; if low-confidence, refine query or search the web. One retry loop.
  • Iterative Agentic RAG (most complex): LangGraph StateGraph; agent retrieves, evaluates, decides to retrieve more or generate. Multiple loop iterations.

⏱️ Understand It in 30 Seconds

One line: Agentic RAG makes retrieval a TOOL the LLM decides to call, enabling query routing, multi-source retrieval, and iterative retrieval.

One analogy:

Standard RAG is like a fixed research procedure: “Step 1: go to the library. Step 2: search for the exact topic. Step 3: write the answer from what you found.” Agentic RAG is like a skilled researcher: she decides whether to go to the library, which of three libraries to use, whether one search is enough or a follow-up search is needed, and whether to combine library research with a web search. The researcher’s JUDGMENT (the LLM) orchestrates the retrieval strategy.

One insight: The architectural shift from “retrieval as a fixed step” to “retrieval as a tool” changes the failure mode. In standard RAG, the system ALWAYS retrieves and the question is “did it retrieve the right context?” In agentic RAG, the agent can decide NOT to retrieve, and the question becomes “did the agent make the right routing decision?” This second failure mode requires agent-specific evaluation (did the routing decision match the ground truth?) in addition to standard RAGAS metrics.

Persona pitches:

Junior Developer: Agentic RAG is LangGraph with a router node. Build a StateGraph with three nodes: classify_query (LLM decides: FAQ, policy, or direct-answer), retrieve_faq, retrieve_policy, generate. Use add_conditional_edges to route based on the route field in the state. The routing node is the only new concept; the retrieval and generation nodes are standard RAG.

Mid-Level Architect: The key design decision is LOOP BUDGET: how many retrieval iterations do you allow? Each iteration adds 500-1,500ms and one LLM call. For most use cases, max_iterations=2 is sufficient and safe. Set max_iterations as a hard limit in the graph state to prevent infinite loops. Monitor loop count distribution in production: if > 30% of queries use max_iterations, the retrieval quality is poor and the loop is not helping; fix the underlying retrieval.

Skeptical Reviewer: In 12 months of production operation, our iterative agentic RAG reduced the cases where users needed to rephrase their question by 40%. But it DOUBLED average latency (from 1.1s to 2.2s) and INCREASED cost by 60% (extra LLM calls for routing + iteration). We ended up A/B testing agentic RAG only for complex multi-part queries (classified upfront) and using standard RAG for simple queries. That hybrid saved 80% of the latency and cost while retaining 90% of the quality improvement.

🔩 First Principles Explanation

Standard RAG treats retrieval as MANDATORY and UNIFORM. Every query goes through the same pipeline regardless of whether retrieval is needed or which source is most relevant. This works when the query distribution is homogeneous and all answers come from one source.

Agentic RAG treats retrieval as CONDITIONAL and ADAPTIVE. The LLM is the orchestrator; it decides the retrieval strategy based on the query. This is more powerful but adds a new dependency: the QUALITY OF THE LLM’S ROUTING DECISION determines retrieval quality.

The core tension: adding LLM reasoning to the orchestration layer makes the system MORE CAPABLE but LESS PREDICTABLE. A fixed pipeline has deterministic execution; an LLM-orchestrated pipeline can take different paths for the same query on different runs (temperature

0). This requires DETERMINISM CONTROLS (temperature=0 for routing decisions; strict output format; structured output parsing).

ESSENTIAL vs ACCIDENTAL COMPLEXITY:

Essential: Deciding WHEN and WHERE to retrieve based on query characteristics is genuinely complex reasoning that a fixed pipeline cannot do. A query that needs two retrieval sources cannot be served by a single vector search.

Accidental: LangGraph’s specific API (StateGraph, add_conditional_edges, compile) is one implementation; the concept of conditional retrieval routing is implementable in any orchestration framework. The API complexity is accidental.

🧪 Thought Experiment

SETUP: A company has three knowledge bases: (1) Product FAQs (10,000 chunks), (2) Legal Policies (5,000 chunks), (3) Engineering Runbooks (8,000 chunks). Queries come in three types: customer questions (60%), compliance questions (20%), DevOps questions (20%).

WITHOUT AGENTIC RAG (standard RAG with single index): All 23,000 chunks in one vector index. A customer question about product pricing retrieves the most similar chunks by embedding; frequently returns a mix of FAQ, policy, and runbook chunks. LLM must synthesize across irrelevant context. RAGAS context_precision = 0.61 (39% of retrieved chunks are from wrong sources). Users report that answers often include irrelevant legal or technical details.

WITH AGENTIC RAG (query router + 3 separate indexes): Routing LLM call (50ms) classifies query type. Customer question -> retrieves ONLY from FAQ index. Compliance question -> retrieves ONLY from Policy index. DevOps question -> retrieves ONLY from Runbook index. Context_precision for each source: 0.89-0.93 (only relevant chunks retrieved). Cost increase: +1 routing LLM call per query (+$0.001/query). Latency increase: +50ms for routing.

THE INSIGHT: For multi-source knowledge bases, even the SIMPLEST agentic pattern (query router with no iteration) provides a large quality improvement at minimal cost. The 90% use case for “agentic RAG” is just query routing; iterative multi-hop retrieval is needed for only 5-10% of queries.

Non-obvious: graph state GROWS with each turn (full message history appended); without state truncation, token count grows unboundedly and eventually exceeds context window. Implement a trim_messages node that runs every N turns, keeping only the last K messages plus the system summary. Graph state serialization: for production, avoid storing embeddings in graph state (they are large and slow state serialization); store retrieval results as chunk IDs + text only.

⚙️ How It Works (Mechanism)

LangGraph StateGraph execution model:

A StateGraph is a directed graph where nodes are Python functions that transform the shared State and edges define the execution path. Conditional edges are functions that return the next node name based on state values.

State accumulation: Each node receives the FULL state dict and returns a PARTIAL state update (dict of keys to update). LangGraph merges updates using operator.add for list fields (messages accumulate) and direct assignment for non-list fields.

Execution: The graph runs as a while True loop: execute current node -> apply state update -> evaluate edges -> select next node. Execution terminates when the current node’s edge points to END.

Loop safety: A loop counter in the state prevents infinite iteration. The counter is incremented in each retrieval node and checked in the conditional edge; if loop_count >= max_iterations, the edge routes to the generation node regardless of context quality.

CONCURRENCY / THREAD-SAFETY:

LangGraph nodes execute SEQUENTIALLY by default. For parallel retrieval (query requires both FAQ and policy in the same graph run), LangGraph supports a fan_out/fan_in pattern: route to two parallel branches that merge at a combine_context node. The fan_out pattern uses LangGraph’s Send API (LangGraph >= 0.2). State updates from parallel branches are merged; list fields (like context) use append merge. Thread-safety: each graph invocation gets its own state copy; the graph object itself is stateless and safe for concurrent invocations. State persistence (SqliteSaver) is NOT thread-safe for concurrent writes to the same thread_id (conversation); use one state store per conversation ID.

WHAT CHANGES AT SCALE:

At 10,000 queries/day: LangGraph with SqliteSaver works; SQLite write serialization is not a bottleneck at this volume. At 100,000 queries/day: migrate to RedisSaver; Redis Cluster for distributed state; route the routing LLM call through a dedicated fast-inference endpoint (gpt-4o-mini, not gpt-4o) to reduce routing latency to 30ms. At 1M queries/day: routing overhead becomes significant; consider training a lightweight classifier (logistic regression or small fine-tuned model) to replace the LLM routing call; eliminates 50ms routing latency and the LLM cost for the routing step. State management: prune conversation states older than 30 days; implement compaction (summarize long conversations to reduce token count in state).

# BAD: iterative retrieval with no loop limit

def retrieve_and_evaluate_bad(state):

# GOOD: explicit loop counter in state
from typing import TypedDict, Annotated
import operator
from langgraph.graph import StateGraph, END

class AgentState(TypedDict):
    query: str
    original_query: str
    context: Annotated[list[str], operator.add]  # accumulate
    iteration: int
    answer: str

MAX_ITERATIONS = 2  # hard limit

def retrieve_node(state: AgentState) -> AgentState:
    """Retrieve context for current query."""
    chunks = retrieve_from_vector_db(state["query"])
    return {
        **state,
        "context": [c["text"] for c in chunks],
        "iteration": state.get("iteration", 0) + 1,
    }

def evaluate_node(state: AgentState) -> AgentState:
    """Evaluate: is context sufficient to answer?"""
    if state["iteration"] >= MAX_ITERATIONS:
        # Budget exhausted; force generate
        return {**state, "sufficient": True}
    # Heuristic: check min similarity score
    context_quality = len([
        c for c in state["context"]
        if c  # non-empty chunks
    ])
    sufficient = context_quality >= 3  # need 3+ chunks
    return {**state, "sufficient": sufficient}

def should_iterate(state: AgentState) -> str:
    """Conditional edge: iterate or generate."""
    if state.get("sufficient", False):
        return "generate"
    return "refine"

def refine_node(state: AgentState) -> AgentState:
    """Refine the query for next iteration."""
    from langchain_openai import ChatOpenAI
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    refined = llm.invoke(
        f"The query '{state['original_query']}' "
        f"returned insufficient context. "
        "Rephrase the query to be more specific. "
        "Reply with ONLY the new query text."
    ).content.strip()
    return {**state, "query": refined}

graph = StateGraph(AgentState)
graph.add_node("retrieve", retrieve_node)
graph.add_node("evaluate", evaluate_node)
graph.add_node("refine", refine_node)
graph.add_node("generate", generate_answer)
graph.set_entry_point("retrieve")
graph.add_edge("retrieve", "evaluate")
graph.add_conditional_edges("evaluate", should_iterate)
graph.add_edge("refine", "retrieve")  # loop back
graph.add_edge("generate", END)
app = graph.compile()

Code walkthrough: BAD (infinite loop risk) vs GOOD (loop budget enforced). Key mechanism: iteration counter in state is incremented by retrieve_node; evaluate_node checks state["iteration"] >= MAX_ITERATIONS FIRST, before the quality check, to guarantee termination. The Annotated[list[str], operator.add] type on context tells LangGraph to APPEND to the list rather than replace it on each iteration (accumulated context from all iterations). Why it matters: without a hard iteration limit, a poorly prompted is_sufficient function can cause infinite loops; these manifest as LLM timeout errors that are difficult to debug. What breaks: MAX_ITERATIONS = 2 is a placeholder; calibrate against production query logs; if > 20% of queries hit max_iterations, the retrieval quality is poor and the loop is not the solution. Takeaway: ALWAYS encode the loop budget as a hard state field, not as a guideline; the graph must guarantee termination.

Scenario 2 - Production: query router with structured output

from pydantic import BaseModel, Field
from langchain_openai import ChatOpenAI
from typing import Literal

class RouteDecision(BaseModel):
    """Structured routing decision."""
    source: Literal["faq", "policy", "direct"] = Field(
        description="best knowledge source for query"
    )
    confidence: float = Field(
        ge=0.0, le=1.0,
        description="routing confidence 0-1",
    )
    reason: str = Field(
        description="one-sentence routing reason",
        max_length=100,
    )

router_llm = ChatOpenAI(
    model="gpt-4o-mini",
    temperature=0,  # deterministic routing
).with_structured_output(RouteDecision)

ROUTE_PROMPT = """
You are a query router for a knowledge management system.
Route this query to the best knowledge source.

Knowledge sources:
- faq: product questions, pricing, features, how-to guides
- policy: legal, compliance, terms of service, data privacy
- direct: math, definitions, factual questions (no lookup needed)

Query: {query}
"""

async def route_query(state: AgentState) -> AgentState:
    """Route query with structured output + fallback."""
    try:
        decision = router_llm.invoke(
            ROUTE_PROMPT.format(query=state["query"])
        )
        route = decision.source
        # Low confidence: fall back to FAQ (broadest source)
        if decision.confidence < 0.65:
            route = "faq"
    except Exception:
        # Routing failure: default to FAQ
        route = "faq"
    return {**state, "route": route}

Code walkthrough: Production query router with Pydantic structured output. Key mechanism: with_structured_output(RouteDecision) makes the LLM return a Pydantic model rather than free text; this eliminates string parsing errors and enforces the Literal["faq", "policy", "direct"] constraint at the type level. The confidence < 0.65 fallback handles ambiguous queries (query could be either FAQ or policy) by defaulting to the broadest source. The except Exception fallback handles routing LLM failures without cascading to the user. Why it matters: in production, routing parse errors (wrong case, unexpected word) cause silent routing failures where queries go to the wrong source; structured output eliminates this class of error entirely. What breaks: with_structured_output uses a second LLM call internally to parse the response (in some LangChain versions); if the parsing fails, it raises OutputParserException. The except Exception catches this. Takeaway: always use structured output for routing in production; never rely on string parsing of LLM responses for control flow.

Key mechanism: thread_id = f"{user_id}:{session_id}" uniquely identifies a conversation; all turns within the conversation share the same checkpoint key, giving the agent memory of previous turns. RedisSaver serializes graph state to Redis; the graph reads its previous state at the start of each turn, enabling context-aware follow-up questions. Why it matters: without persistence, every query is stateless (standard RAG); with Redis-backed state, the agent can answer “As I mentioned before…” and build on prior turns. What breaks: SqliteSaver is NOT safe for multi-instance deployments (SQLite write serialization blocks concurrent state updates for the same thread_id); always use RedisSaver in production with horizontal scaling. Takeaway: use RedisSaver for production multi-turn agents; implement state compaction to prevent context window overflow.

🔁 Flow / Lifecycle

Agentic RAG development lifecycle:

  1. START: Validate core retrieval quality before adding agentic complexity. If RAGAS context_recall < 0.75 with standard RAG, fix retrieval first. Adding an agent layer to poor retrieval creates a complex system with poor retrieval.
  2. ADD QUERY ROUTER: Implement Level 1 (query router) if the knowledge base has multiple distinct source types. Validate routing accuracy with 100 test queries; routing accuracy should be > 90%.
  3. MEASURE ROUTING GAP: Run production queries for 2 weeks; identify queries where routing confidence < 0.7; update prompt with examples from these queries.
  4. ADD ITERATION (if needed): Review query logs for multi-hop queries. If < 5% require iteration, the standard router is sufficient. Add Level 3 only when production evidence justifies the latency cost.
  5. ADD PERSISTENCE (if multi-turn required): Implement RedisSaver for multi-turn conversations; add state compaction.
  6. MONITOR: Track routing accuracy, loop count distribution, routing confidence, RAGAS metrics, latency per route path.

💎 Transferable Wisdom

Reusable Principle: Adding intelligence (an LLM) to the ORCHESTRATION LAYER of a system creates CONDITIONAL EXECUTION paths that are more powerful but harder to monitor. Whenever the orchestrator is non-deterministic (an LLM), you must monitor the ORCHESTRATION DECISIONS (routing confidence, iteration counts) as first-class metrics, not just the final output quality.

Where Else This Applies:

  1. Business process orchestration: A workflow engine (Temporal, Camunda) that routes work items based on business rules is the same pattern as a query router. Adding an LLM to classify work items (instead of explicit rules) creates the same monitoring challenge: classification decisions must be logged and evaluated, not just work item outcomes.
  2. AI customer service routing: Routing customer service requests to the correct team (billing, technical, sales) via LLM classification is identical to query routing in agentic RAG. The same principles apply: temperature=0 for determinism; structured output; confidence threshold; fallback; logging of routing decisions.
  3. Autonomous driving decision planning: A planning agent decides whether to slow down, change lanes, or maintain speed. Monitoring the DECISION (not just the outcome) is critical for safety; similarly, monitoring ROUTING DECISIONS in agentic RAG is critical for quality.

Industry Applications: Enterprise knowledge management, multi-domain customer service automation, autonomous research agents, intelligent document processing pipelines.

💡 The Surprising Truth

The surprising truth about agentic RAG is that most production systems labeled “agentic RAG” are NOT using iterative retrieval. They are query routers with conditional branching - essentially a switch statement implemented with an LLM call. The term “agentic” creates the expectation of an agent that autonomously decides retrieval strategy across multiple iterations; the reality is usually one LLM call that picks a bucket (FAQ vs policy vs direct-answer), followed by a standard RAG retrieval and generation.

The reason iterative agentic RAG (true multi-hop retrieval in a loop) is rare in production: at $0.01-$0.03 per LLM call (routing + evaluation

  • generation), a 3-iteration retrieval loop costs $0.06-$0.12 per query. At 100,000 queries/day: $6,000-$12,000/day. Compare to standard RAG at $0.01/query: $1,000/day. The quality improvement from iterative retrieval on the 5% of genuinely multi-hop queries does not justify 6x the cost on ALL queries.

The production pattern that actually works: classify queries upfront using query complexity (simple=<30 tokens, complex=>30 tokens + multiple entities detected); route complex queries to iterative agentic RAG; route simple queries to standard RAG. This hybrid applies the expensive pattern only where it helps, reducing the cost premium to 10-15%.

🧠 Think About This Before We Continue

Q1: Why is monitoring routing CONFIDENCE more important than monitoring routing ACCURACY in production?

Hint: Routing accuracy requires ground-truth labels (“this query should have gone to policy”), which requires manual annotation. Routing confidence is produced by the routing LLM itself (structured output includes confidence: float) and is available for every query without labels. High confidence

  • correct route (normal). Low confidence + correct route (lucky; prompt needs examples). Low confidence
  • wrong route (problem). The trend in routing confidence is a LEADING INDICATOR of routing accuracy degradation; it detects drift before wrong answers appear. By monitoring confidence trends, you can identify when the routing prompt needs updating before users notice quality drops.

Q2: LangGraph’s operator.add annotation on context accumulates context from all iterations. What is the failure mode if context accumulates indefinitely, and how does it interact with the loop budget?

Hint: If context: Annotated[list[str], operator.add] accumulates context from all iterations and MAX_ITERATIONS=5, the final generation step receives context from 5 retrieval passes (potentially 5 x k chunks = 50 chunks total). 50 chunks x 500 tokens = 25,000 tokens of context. This may exceed the LLM’s context window (gpt-4o-mini: 128K; but 25K tokens is expensive) and reduces generation quality (too much context confuses the LLM). Mitigation: after the evaluate node, REPLACE context with refined context (not add): use direct assignment instead of operator.add for the refinement retrieval. Only accumulate context when multi-source fan-out requires combining contexts; for iterative single-source retrieval, replace context on each iteration.

What fraction falls into each category? Based on this analysis, which agentic RAG level is justified?

Hint: Expected distribution for most enterprise knowledge base applications: (a) 70-80% simple single-hop, (b) 10-15% multi-source, (c) 2-5% multi- hop, (d) 10-20% direct-answer. This distribution suggests Level 1 (query router) handles 25-35% of queries better than standard RAG; Level 3 (iterative) helps only 2-5%. If multi-hop queries are < 5%, Level 3 is hard to justify on cost grounds. If multi-hop queries are > 15%, the use case benefits from Level 3 and the cost premium is justifiable. The analysis should drive architecture, not assumptions.

🎯 Interview Deep-Dive

Q1: When would you choose agentic RAG over standard RAG for a production system?

Expected answer: Three scenarios justify agentic RAG: (1) Multiple knowledge sources - if the system has 2+ distinct knowledge bases with low overlap (FAQ vs legal policies vs technical runbooks), a query router improves context_precision by 30-50% at +$0.001/query. (2) Multi-hop queries - if query logs show > 5-10% of queries require information from two sources to answer correctly, iterative retrieval with a 2-iteration budget improves quality. (3) Direct-answer bypass - if 15-20% of queries can be answered without retrieval (mathematical questions, factual lookups, greetings), skipping retrieval for these reduces cost and latency. Standard RAG is preferred when: single homogeneous source, simple query distribution, hard P95 latency requirement (< 1 second), cost sensitivity. The decision requires query log analysis, not assumptions.

Q2: How do you prevent infinite loops in a LangGraph iterative retrieval agent?

Expected answer: Three layers of protection: (1) State counter: iteration: int in graph state, incremented in the retrieve node, checked FIRST in the evaluate node before the quality check. if iteration >= MAX_ITERATIONS: return sufficient=True. This is the hard limit. (2) LangGraph recursion limit: graph.compile(recursion_limit=10); LangGraph raises an exception if the graph executes more than N nodes; this is a backstop for bugs in the iteration counter logic. (3) Async timeout: wrap app.ainvoke() in asyncio.wait_for(app.ainvoke(...), timeout=30.0); if the graph runs for more than 30 seconds (e.g., due to slow LLM calls on each iteration), the timeout fires and a graceful error is returned. All three layers together ensure the system cannot loop indefinitely regardless of edge cases in the iteration counter.

Q3: Walk me through the state management design for a multi-turn agentic RAG conversation at 100,000 users/day.

Expected answer: Requirements: (1) Persistence across turns (user expects agent to remember prior questions). (2) Concurrent access (multiple users simultaneously). (3) State growth control (conversation history grows each turn). Design: Use RedisSaver (not SqliteSaver) with Redis Cluster for horizontal scaling; SqliteSaver has global write serialization unsuitable for concurrent use. Key-space design: state:{user_id}:{session_id}. TTL: 24 hours of inactivity (Redis EXPIRE command). State compaction: every 10 turns, run a compact_state node that summarizes the conversation history into a 200-word summary and replaces the message list with [system_prompt

  • summary_message + last_3_messages]. This keeps the state token count bounded. For 100K users/day with 5 turns average: peak concurrent state reads = ~5,000/second. Redis Cluster with 3 shards handles this comfortably. Storage: 100K sessions x 2KB average state = 200MB; negligible.

Q4: A production agentic RAG system’s routing accuracy has dropped from 92% to 78%. How do you diagnose and fix this?

Expected answer: Three-step diagnosis: (1) Pull routing logs for the past 2 weeks; filter for queries with routing confidence < 0.75; examine the query text for patterns. Likely finding: a new category of queries (e.g., a new product launch introduced queries about “subscription tiers” that don’t clearly map to FAQ or Policy). (2) Run a confusion matrix on hand-labeled samples: which source categories are being confused? (3) Calculate routing confidence distribution trend; if P25 confidence dropped from 0.85 to 0.62, the prompt is mis-matched to the new query distribution. Fix: add 5-10 new examples to the routing prompt from the misclassified query category. Use with_structured_output to ensure the new examples are in the correct format. Re-evaluate routing accuracy on 100 test queries. If accuracy does not recover to > 90% with prompt engineering, consider fine-tuning a small routing model on the labeled query log; a fine-tuned bert-base-uncased classifier with 500 examples achieves > 95% routing accuracy at 5ms latency (vs 50ms for LLM routing).

Q5: How do you evaluate an agentic RAG system? What metrics does standard RAGAS miss?

Expected answer: RAGAS measures RETRIEVAL and GENERATION quality: context_recall, context_precision, answer_faithfulness, answer_relevancy. What RAGAS MISSES for agentic systems: (1) ROUTING ACCURACY: what percentage of queries were routed to the correct source? Requires a hand-labeled test set of 100+ queries with ground-truth source labels. (2) ITERATION EFFICIENCY: what fraction of queries used 0, 1, 2, N iterations? If > 20% use max_iterations, the iteration loop is not converging; the underlying retrieval quality is poor. (3) BYPASS CORRECTNESS: what fraction of direct-answer bypasses were correct (no retrieval needed)? And what fraction were INCORRECT bypasses (agent said no retrieval needed but retrieval would have helped)? (4) LOOP CONTRIBUTION: for iterated queries, did the second retrieval actually improve the answer (measured by faithfulness delta)? If faithfulness does not improve after iteration, the loop is adding cost without benefit. Full evaluation suite: RAGAS (standard) + routing accuracy (labeled test set) + iteration efficiency (production metrics)

  • bypass correctness (sampled review).

Q6: What is Self-RAG and when would you use it over a LangGraph-based agent?

Expected answer: Self-RAG is a fine-tuned model that learns to INTERLEAVE retrieval decisions with generation: the model outputs special tokens (“Retrieve”, “Isrel”, “Issup”, “Isuse”) that indicate when to retrieve and how to evaluate retrieved context. The retrieval decision is made MID-GENERATION rather than as a separate orchestration step. Advantages over LangGraph-based agents: (1) No separate router LLM call (zero latency for routing); (2) Retrieval decision is coherent with generation context (the model knows exactly what it has generated so far when it decides to retrieve); (3) Single inference call (no orchestration framework overhead). Disadvantages: (1) Requires a fine-tuned model (OpenAI/Claude API models cannot be fine-tuned for Self-RAG tokens); (2) Tied to a specific model; cannot swap model easily; (3) Fine-tuning requires labeled data. Use Self-RAG when: you have a dedicated model deployment (Llama or similar open-source), you need minimal latency, and you have labeled training data. Use LangGraph-based agents when: you use API-based models (GPT-4o, Claude), you need multi-source routing, or you need multi-turn persistence.

🌍 Sustainability & Ethics

Environmental Impact:

Agentic RAG increases the number of LLM inference calls per query (routing + optional evaluation

  • generation = 2-4 calls minimum). At 100K queries/day with 3 calls average: 300K LLM inferences/day vs 100K for standard RAG. Energy cost: 3x the standard RAG baseline. Optimization: use smaller models for routing (gpt-4o-mini or a fine-tuned classifier); a logistic regression routing classifier uses 1/1,000 of the energy of a GPT-4o routing call. State persistence in Redis requires always-on infrastructure; Redis Cluster adds ~10% to the energy footprint vs stateless standard RAG. Recommendation: route to smaller models wherever possible; reserve large models for generation only.

Economic Reality:

Query router: +$0.001/query x 100K/day = $100/day additional cost. Iterative with 2 iterations average: +$0.02/query x 10K complex queries/day = $200/day. Redis state management: $150/month for moderate scale. At 100K queries/day, agentic RAG premium: $100-$200/day = $3,000-$6,000/month vs standard RAG at $1,000/day. Total agentic RAG cost at this scale: $4,000-$7,000/day. Justification: if routing reduces support ticket escalation rate by 20% (queries answered correctly without human intervention), the savings in support cost often exceed the agentic premium.

Ethical Implications:

Multi-turn stateful agents accumulate conversation history that may include sensitive user disclosures (health concerns, financial situations, personal decisions). This history is stored in Redis with user-linked keys; it is subject to GDPR data subject access and erasure rights. Conversation state must have a retention policy (e.g., delete after 30 days or on session end). The routing decision (which source the agent accessed) must be auditable for high-risk domains (the agent accessed the legal policies database on behalf of user X at time T). Bias in routing: if the routing prompt’s examples reflect the demographics of the team that wrote them, queries phrased in non-standard English may have lower routing confidence and higher error rates.

📋 Postmortem Triggers

Trigger 1: Any query’s total execution time exceeds 30 seconds.

Why postmortem: A 30-second query indicates a runaway loop or a stuck LLM call. Even if the query eventually resolves, a 30-second response time violates every reasonable SLO and indicates a missing safeguard (loop budget, timeout, circuit breaker). What could have gone worse: without the 30-second timeout, the query could hang indefinitely, eventually exhausting the thread pool and taking down the entire service. The postmortem must confirm that the three layers of loop protection (state counter, recursion_limit, async timeout) are all in place after the incident.

Trigger 2 (silent failure): Routing confidence P50 drops below 0.75 and remains there for more than 4 hours WITHOUT any alert firing.

Why postmortem: If routing confidence degraded silently for 4 hours, it means: (a) the routing confidence metric was not being monitored; or (b) the alert threshold was set too low. The degradation caused incorrect routing, which degraded context_precision, which degraded answer quality - all without a single error rate alert. The RAGAS quality alert should have fired before the 4-hour mark. The postmortem must review the alerting configuration and add a routing confidence P50 alert threshold at 0.78 (warning) and 0.70 (critical).

Trigger 3: A user reports that the agent accessed the “legal policies” knowledge source in response to a simple product question (wrong routing confirmed from logs).

Why postmortem: This is a routing error with a user-visible impact. Even one confirmed wrong-source routing is significant in a legal RAG system because it may have presented inapplicable legal information as relevant policy. The postmortem must: (a) check how many similar queries were affected (look for routing confidence < 0.75 on the query pattern); (b) determine if the LLM’s wrong routing was caused by an ambiguous query phrasing or a prompt gap; (c) check if the user made any decisions based on the incorrectly sourced response.
