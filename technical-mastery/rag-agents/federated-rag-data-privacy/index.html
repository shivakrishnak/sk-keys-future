TL;DR: Federated RAG is a retrieval architecture where multiple organizations each maintain their own private RAG index; queries are distributed to each shard; results are aggregated without the raw documents ever leaving the organization’s control. This is NOT federated learning (training a model across distributed data); it is FEDERATED RETRIEVAL (each organization’s index is queried independently; only ranked result SUMMARIES or embedding similarity scores are shared, not the underlying documents). The privacy guarantee depends on what is shared: sharing document chunks verbatim (with content masking) provides weak privacy; sharing only embedding similarity scores (without content) provides stronger privacy but enables inference attacks; sharing only binary signals (does this query match anything in my index?) provides the strongest privacy guarantee but the lowest retrieval quality. The fundamental trade-off in federated RAG: PRIVACY (how much information about documents is revealed to other parties) vs RETRIEVAL QUALITY (how good is the final answer when only partial document signals are shared). Production federated RAG systems manage this trade-off by applying differential privacy to similarity scores (adding calibrated noise before sharing) and by using secure multi-party computation (SMPC) for aggregation in regulated industries (healthcare, financial services) where even similarity scores constitute sensitive information.

🔥 The Problem This Solves

Organizations in regulated industries frequently need to provide AI-powered answers that draw on knowledge distributed across multiple organizations, but cannot centralize that knowledge for RAG due to legal or contractual constraints:

  1. HEALTHCARE: Hospital A has patient records; Hospital B has clinical trial data; Hospital C has specialist consultation notes. A clinician’s query could benefit from all three sources. Centralizing them violates HIPAA (patient data cannot leave the originating organization without a specific data sharing agreement that authorizes RAG-style access).

  2. FINANCIAL SERVICES: Bank A and Bank B are in a consortium for fraud detection. Both have transaction data that would benefit a shared RAG system for fraud pattern queries. But the transaction data is covered by banking secrecy laws; sharing it creates competitive and regulatory risk.

  3. GOVERNMENT / INTELLIGENCE: Agency A and Agency B each have classified document corpora. A query may benefit from both but neither can share its documents with the other at the document level; only declassified answers can cross the boundary.

  4. ENTERPRISE CONSORTIA: Industry consortia (e.g., pharmaceutical companies sharing pre-competitive research) want to enable RAG across member data without any member being able to read another member’s documents.

In all four cases, the need is: a user query should be answered using knowledge distributed across multiple private corpora, without those corpora ever being centralized.

📘 Textbook Definition

Federated RAG is a retrieval-augmented generation architecture where multiple data custodians each maintain an independent, private RAG index; an orchestrator distributes user queries to each custodian’s retrieval endpoint; custodians return retrieval results (at varying levels of privacy-preservation); the orchestrator aggregates results and provides them to an LLM for generation; raw documents never leave the custodian’s control.

Four federated RAG privacy tiers:

  1. TIER 0 (Verbatim Chunks): Each shard returns matching document chunks as plaintext. HIGH retrieval quality; ZERO document-level privacy (the orchestrator reads all documents). Suitable for: enterprise internal federations where all parties trust the orchestrator.

  2. TIER 1 (Content-Masked Summaries): Each shard returns a generated summary of matching documents; no verbatim content shared. MEDIUM quality; MEDIUM privacy (summary may leak sensitive information).

  3. TIER 2 (DP Similarity Scores): Each shard returns only the top-K similarity scores (with calibrated differential privacy noise added) for the query. The orchestrator uses these scores to decide which shards have relevant content; the orchestrator then makes a targeted request to only the relevant shards. MEDIUM-LOW quality; HIGH privacy.

  4. TIER 3 (SMPC Aggregation): All shards collaboratively compute the aggregated retrieval result using secure multi-party computation; no single party (not even the orchestrator) learns the individual shard contributions. MEDIUM quality; VERY HIGH privacy; HIGH computational overhead.

Key distinction from federated learning:

  • Federated learning: Trains a MODEL across distributed data; shares model GRADIENTS (not raw data) during training.
  • Federated RAG: Retrieves DOCUMENTS at query time across distributed indices; shares retrieval RESULTS (not documents) during inference.
  • These are complementary; federated RAG uses the INFERENCE privacy pattern; federated learning uses the TRAINING privacy pattern.

⏱️ Understand It in 30 Seconds

One line: Federated RAG lets multiple organizations each contribute to answering a query from their private document corpus without any organization’s documents leaving their system; only retrieval SIGNALS (ranked scores, summaries, or SMPC-aggregated vectors) cross organizational boundaries.

One analogy:

Imagine asking a question to five independent librarians, each with access to a different private library. Each librarian looks up the answer in their own library and tells you “yes, I found something relevant” or provides a summary of what they found. None of them lets you read the original books; none of them can see the other librarians’ books. You receive partial answers from each librarian and synthesize a final answer yourself. The privacy guarantee is that no single librarian’s books are exposed to you or to the other librarians - only their search signals are shared.

One insight: The privacy guarantee in federated RAG is not binary: it exists on a spectrum from “almost no privacy” (sharing verbatim chunks) to “strong privacy” (SMPC aggregation). Every tier has a quality trade-off. Most production federated RAG systems choose TIER 1 (content-masked summaries) because it provides a practical balance: legal teams can review the exact content being shared (a summary), it provides enough information for good LLM generation, and it satisfies most regulatory frameworks that prohibit sharing raw records but permit sharing derived information.

Persona pitches:

Junior Developer: Federated RAG is primarily an orchestration problem: how do you send a query to multiple independent retrieval services and aggregate the results? Start with the simplest architecture: a coordinator service that queries each shard’s retrieval API in parallel, collects ranked results, re-ranks them, and sends the top-K to the LLM. The privacy level is a product/legal decision, not a technical default; implement the lowest-overhead option first (Tier 0 or Tier 1) and upgrade privacy tier only if the legal team requires it.

Mid-Level Architect: The key architectural decisions in federated RAG: (1) QUERY DISTRIBUTION - fan-out (query all shards) vs routing (predict which shards are relevant and query only those); fan-out has latency proportional to the slowest shard; routing requires a shard routing model; (2) RESULT AGGREGATION

  • score normalization across shards (each shard scores in its own embedding space; scores are not directly comparable; use relative rank fusion (RRF) instead of raw scores); (3) PRIVACY TIER SELECTION - this is a legal + technical co-decision; Tier 2 (DP scores) requires calibrating the privacy budget (epsilon) which requires knowing the sensitivity of the data.

Skeptical Reviewer: We built a Tier 1 federated RAG for a healthcare consortium (4 hospitals). The pilot was approved. Three months post-launch, a privacy auditor found that the “summaries” generated by Hospital A’s shard included patient ages, hospital locations, and treatment outcomes (the LLM-generated summaries were not sufficiently anonymized; they described cases in enough detail to re-identify patients). The lesson: “summary” does not automatically mean “private”; the content-masking function at each shard must have an explicit set of PII categories to redact; LLM-generated summaries require PII detection before sharing; the privacy guarantee must be verified by a privacy auditor, not assumed.

🔩 First Principles Explanation

Federated RAG is built on two fundamental properties of the retrieval process that enable privacy:

  1. SEPARABILITY OF RELEVANCE FROM CONTENT: A retrieval system can answer “is document D relevant to query Q?” (relevance signal) without sharing the content of document D. The relevance signal is a derived quantity (a similarity score or a rank); the content is the raw data. Privacy-preserving retrieval shares the relevance signal while protecting the content.

  2. DIMINISHING INFORMATION IN AGGREGATION: The final answer quality depends on which documents are MOST relevant (the top-K); it does not require knowing the SCORES of all other documents. This means a shard can share only its top-K results (ranked, not scored) without revealing the full score distribution across its corpus.

The privacy risk arises from two attack vectors:

  • CONTENT INFERENCE from signals: Even without sharing document content, sharing similarity scores can reveal information about documents. If a query is “patient John Smith, diagnosis 2023” and the shard reports a high similarity score, the attacker infers that patient John Smith’s records exist in that corpus. This is a membership inference attack. Differential privacy noise on scores defends against this.

  • RECONSTRUCTION from summaries: If the shard returns a summary of matching documents, a sufficiently detailed summary can allow reconstruction of the original document. Content-masking (PII removal, detail abstraction) reduces this risk but does not eliminate it for uniquely-identifiable cases.

The choice of privacy tier is a choice about which attack vector to defend against and at what cost to retrieval quality.

ESSENTIAL vs ACCIDENTAL COMPLEXITY:

Essential: The tension between retrieval quality (which requires sharing specific content) and privacy (which requires limiting content sharing) is fundamental to federated RAG; no architecture eliminates this tension without accepting either quality loss or privacy loss.

Accidental: The high operational overhead of SMPC for aggregation in Tier 3 is partly accidental; advances in secure computation hardware and protocols continue to reduce this overhead; the 2025 state-of-the-art SMPC overhead for federated retrieval is 3-5x baseline latency (down from 20x in 2022).

🧪 Thought Experiment

SETUP: A pharmaceutical consortium wants to enable federated RAG across 5 member companies’ drug discovery databases. Each company has 500,000 proprietary documents. No company’s documents can be shared with any other. A researcher’s query: “What is known about the efficacy of compound X for target Y?”

WHAT HAPPENS WITHOUT FEDERATED RAG: Each company deploys its own RAG system; researchers must query 5 separate systems and manually synthesize the answers. Quality degrades because no single answer draws on all 5 corpora. Each researcher needs access to all 5 systems, creating authentication complexity. If any company accidentally grants too-broad access, another company’s documents are exposed.

WHAT HAPPENS WITH FEDERATED RAG (TIER 1): The consortium deploys an orchestrator. The researcher’s query is distributed to all 5 shards. Each shard retrieves its top-3 matching documents and generates an anonymized summary that: (a) replaces all proprietary compound names with generic descriptors, (b) removes company identifiers, (c) retains scientific findings. The orchestrator receives 15 summaries, re-ranks them by relevance, sends the top-5 to the LLM for synthesis. The LLM generates a synthesized answer citing “multiple consortium sources” without identifying which company contributed which finding. Privacy preserved: no company’s verbatim documents are shared; company attribution is hidden. Quality: the researcher gets a synthesized answer drawing on all 5 corpora.

THE INSIGHT: Federated RAG is not a perfect privacy solution; it is a practical engineering compromise that moves the privacy risk from “documents are exposed” to “summaries may leak information.” The acceptable privacy tier depends on the regulatory framework and the sensitivity of the specific information. The thought experiment reveals the core design question: “What is the minimum information that must be shared to produce a useful answer?” Design around that minimum.

📶 Gradual Depth - Five Levels

Level 1 (Child): Federated RAG means asking a question to several libraries that each have their own books and cannot share them. Each library looks for the answer in its own books and sends you a hint about what it found, without sending you the actual book pages. You use all the hints together to write your answer.

Level 2 (Junior): Multiple organizations each have private document indices. A query is sent to all of them in parallel. Each returns its top-K results (as summaries or scores, not raw documents). An orchestrator aggregates and re-ranks the results using Reciprocal Rank Fusion (RRF), then sends the top-K context to an LLM for generation. Raw documents never leave each organization.

Level 3 (Mid): Four privacy tiers: Tier 0 (verbatim chunks: high quality, no privacy), Tier 1 (content-masked summaries: medium quality, medium privacy), Tier 2 (DP similarity scores: low-medium quality, high privacy), Tier 3 (SMPC aggregation: medium quality, very high privacy, high overhead). Re-ranking across shards uses RRF (not raw score comparison) because each shard’s embedding space is independent. Differential privacy noise (Laplace mechanism) on scores calibrated to sensitivity of the corpus.

Level 4 (Senior): Technical details: RRF formula is RRF(d) = sum(1/(k + r(d))) where r(d) is the rank of document d in each shard’s result list and k=60 is a constant that dampens the impact of high-ranked documents; RRF is more stable than score-normalized fusion because it does not require calibrated inter- shard score comparison. For Tier 2 differential privacy: the privacy budget epsilon must be calibrated for each query type; a high-sensitivity query (specific named individual) requires smaller epsilon (more noise) than a low-sensitivity query (general topic); this requires a query sensitivity estimator at the orchestrator. For SMPC in Tier 3: computational overhead is 3-10x baseline; typically implemented using additive secret sharing (each shard sends a secret share of its vector; the orchestrator reconstructs the aggregated result without learning individual shard contributions).

Level 5 (Expert Cues): Non-obvious challenges: (1) SHARD HETEROGENEITY: each shard’s embedding model may differ (different versions, different fine-tuning); a query embedded by Model A may not be comparable to document embeddings from Shard B’s Model B; solution: agree on a shared query embedding model (all shards use the same query encoder for scoring, even if their document encoder differs); (2) MEMBERSHIP INFERENCE DEFENSE: even with DP noise, a sustained attacker can run many queries about a specific entity and average out the noise to recover the membership signal; defense: rate limiting per entity per shard + per-query DP budget accounting (global epsilon tracking, not per- query); (3) SHARD COLLUSION: if two shards collude and share their scores, they can reconstruct the full score distribution; SMPC is the only defense against colluding shards; Tier 1/2 defenses assume non-colluding parties; (4) ANSWER ATTRIBUTION LEAKAGE: even if raw documents are not shared, the generated answer may implicitly reveal which shard contributed (if the answer describes a fact only known to one shard, an attacker can infer shard membership); mitigate with answer grounding constraints (cite “consortium sources” without identifying which shard).

⚙️ How It Works (Mechanism)

Federated query processing pipeline:

  1. QUERY EMBEDDING: Orchestrator embeds the user query using the AGREED shared query encoder (all shards have agreed to use the same query embedding model to ensure cross-shard comparability).

  2. FAN-OUT: Orchestrator sends the query embedding (and original query text, for summary generation) to all shards in parallel, with a shard-level timeout.

  3. SHARD-LOCAL RETRIEVAL: Each shard performs ANN search against its local vector index; retrieves top-K results.

  4. PRIVACY FILTERING (at shard): Each shard applies its configured privacy tier:
    • Tier 0: return verbatim chunks
    • Tier 1: run LLM summarizer + PII filter; return summary
    • Tier 2: add Laplace noise to scores; return (rank, noisy_score) pairs
    • Tier 3: participate in SMPC protocol; contribute secret share

  5. AGGREGATION (at orchestrator): Collect shard responses; apply RRF for result fusion; form top-K context.

  6. GENERATION: Send top-K context to LLM; generate response.

CONCURRENCY / THREAD-SAFETY:

Fan-out to N shards is inherently concurrent (parallel HTTP calls or gRPC streams to shard endpoints). Critical concurrency concerns: (1) SHARD TIMEOUT: use a configurable timeout per shard (e.g., 500ms for low-latency, 2000ms for batch mode); shard timeouts must not block the entire query; use asyncio.gather(return_exceptions=True) or equivalent; (2) RESULT ORDERING: RRF aggregation must be deterministic given the same inputs; results from concurrent shard calls arrive in non-deterministic order; buffer all results before RRF; (3) DP BUDGET ACCOUNTING: if the DP privacy budget is tracked per-query globally (to prevent noise averaging attacks), the budget accounting must be serialized (not concurrent) to prevent race conditions on the budget counter.

Scenario 1 - BAD vs GOOD: score-based vs rank-based cross-shard aggregation

# Cross-shard result aggregation in Federated RAG

# BAD: aggregating raw similarity scores across shards
def aggregate_by_raw_score(shard_results: dict[str, list]) -> list:
    """
    BAD: Averaging or comparing raw similarity scores from different shards.
    Problem: Each shard has its own embedding model + normalization.
    Shard A's score of 0.85 is NOT comparable to Shard B's 0.85.
    Shard A might use cosine similarity; Shard B might use dot product.
    Shard A might have 10K documents; Shard B might have 1M documents.
    Result: shards with higher average scores (more verbose docs)
    dominate the aggregation; smaller shards with high-quality
    matches are ranked below lower-quality matches from larger shards.
    """
    all_results = []
    for shard_id, results in shard_results.items():
        for doc in results:
            all_results.append({
                "shard": shard_id,
                "content": doc["content"],
                "score": doc["score"],  # INCOMPARABLE across shards
            })
    # Sorting by raw score: WRONG (apples to oranges comparison)
    return sorted(all_results, key=lambda x: x["score"], reverse=True)


# GOOD: Reciprocal Rank Fusion (RRF) for cross-shard aggregation
def aggregate_by_rrf(
    shard_results: dict[str, list[dict]],
    k: int = 60,  # RRF constant (60 is empirically robust)
) -> list[dict]:
    """
    GOOD: Reciprocal Rank Fusion for cross-shard aggregation.
    Operates on RANKS (1, 2, 3...) not raw scores.
    Ranks ARE cross-shard comparable: rank 1 means "best match
    in this shard" regardless of the underlying scoring system.
    
    RRF score: sum(1 / (k + rank)) for each shard
    Higher RRF score = appears high-ranked in more shards.
    """
    # Map document identifier to its RRF score
    rrf_scores: dict[str, float] = {}
    doc_registry: dict[str, dict] = {}

    for shard_id, results in shard_results.items():
        for rank_0indexed, doc in enumerate(results):
            rank = rank_0indexed + 1  # 1-based rank
            doc_id = doc.get("doc_id") or f"{shard_id}_{rank}"

            # Add RRF contribution for this shard
            rrf_scores[doc_id] = (
                rrf_scores.get(doc_id, 0.0) + 1.0 / (k + rank)
            )

            # Store document content (first shard wins for content)
            if doc_id not in doc_registry:
                doc_registry[doc_id] = {
                    "doc_id": doc_id,
                    "content": doc["content"],
                    "source_shard": shard_id,
                    "original_rank_in_shard": rank,
                }

    # Sort by RRF score (higher = more relevant across shards)
    ranked_docs = sorted(
        rrf_scores.items(), key=lambda x: x[1], reverse=True
    )

    return [
        {**doc_registry[doc_id], "rrf_score": score}
        for doc_id, score in ranked_docs
    ]

Code walkthrough: Raw score vs RRF aggregation. Key mechanism: RRF operates on RANKS (1, 2, 3…) rather than raw similarity scores; ranks are normalized by definition (rank 1 = best in this shard) and are cross- shard comparable. The RRF formula 1 / (k + rank) dampens the advantage of being ranked #1 vs #2 (the difference between rank 1 and rank 2 is smaller with k=60 than without the constant); this prevents a single shard’s top result from dominating over moderate-rank results from multiple other shards. Why it matters: the BAD approach silently introduces shard size bias; larger shards (more documents, higher average query-document overlap) systematically outscore smaller specialized shards even when the smaller shard has the most relevant document. What breaks: if a document appears in multiple shards (rare but possible in consortium settings), the doc_id must be a content hash, not a shard-local ID; otherwise the same document is counted twice in RRF. Takeaway: cross-shard aggregation always requires rank- based fusion; never compare raw similarity scores from different vector indices.

Scenario 2 - Production: differential privacy for shard similarity scores

import numpy as np
from dataclasses import dataclass
from typing import Optional

@dataclass
class DPScore:
    """
    Differential privacy noise applied to similarity score.
    Using Laplace mechanism: noise ~ Laplace(0, sensitivity/epsilon).
    """
    original_rank: int
    noisy_score: float
    epsilon: float     # Privacy budget consumed
    sensitivity: float # Data sensitivity (query-specific)


class ShardPrivacyFilter:
    """
    Applies differential privacy to shard similarity scores
    before returning to orchestrator.

    Privacy guarantee: for any two datasets D1, D2 differing
    in one document, the probability ratio of the output
    is bounded by exp(epsilon).
    Lower epsilon = stronger privacy = more noise = lower quality.
    """

    QUERY_SENSITIVITY_MAP = {
        "high": 0.1,   # Named individuals, specific medical records
        "medium": 1.0, # General topic queries
        "low": 5.0,    # Aggregate/statistical queries
    }

    def __init__(
        self,
        default_epsilon: float = 1.0,
        max_epsilon_per_day: float = 10.0,
        privacy_budget_tracker: Optional["BudgetTracker"] = None,
    ):
        self.default_epsilon = default_epsilon
        self.max_epsilon_per_day = max_epsilon_per_day
        self.budget_tracker = privacy_budget_tracker

    def apply_dp_noise(
        self,
        scores: list[float],
        query_sensitivity: str = "medium",
        query_id: str = "",
    ) -> list[DPScore]:
        """
        Apply Laplace noise to similarity scores.
        
        Args:
            scores: raw similarity scores from ANN search (sorted desc)
            query_sensitivity: sensitivity level of the query
            query_id: for per-query budget tracking
        
        Returns: list of DP-noised scores (original_rank + noisy_score)
        """
        sensitivity_value = 1.0  # L1 sensitivity of similarity score
        epsilon = self.QUERY_SENSITIVITY_MAP.get(
            query_sensitivity, self.default_epsilon
        )

        # Check remaining privacy budget
        if self.budget_tracker:
            remaining = self.budget_tracker.remaining_budget(query_id)
            if remaining < epsilon:
                # Degrade to higher noise (smaller effective epsilon)
                epsilon = max(remaining, 0.01)

        scale = sensitivity_value / epsilon
        dp_scores = []

        for rank_0indexed, score in enumerate(scores):
            # Laplace noise
            noise = np.random.laplace(loc=0.0, scale=scale)
            noisy_score = float(score + noise)
            # Clip to valid range (similarity scores in [-1, 1])
            noisy_score = max(-1.0, min(1.0, noisy_score))

            dp_scores.append(DPScore(
                original_rank=rank_0indexed + 1,
                noisy_score=noisy_score,
                epsilon=epsilon,
                sensitivity=sensitivity_value,
            ))

            if self.budget_tracker:
                self.budget_tracker.consume(query_id, epsilon)

        return dp_scores

    def filter_to_tier2(
        self,
        retrieval_results: list[dict],
        query_sensitivity: str = "medium",
        top_k: int = 10,
    ) -> list[dict]:
        """
        Convert raw retrieval results to Tier 2 DP output.
        Returns: rank + noisy score ONLY (no content, no doc IDs).
        """
        # Extract scores from top-K results only
        scores = [r["score"] for r in retrieval_results[:top_k]]
        dp_scores = self.apply_dp_noise(
            scores, query_sensitivity=query_sensitivity
        )

        # Return ONLY rank and noisy score (NOT content, NOT doc IDs)
        return [
            {
                "rank": dp.original_rank,
                "noisy_score": dp.noisy_score,
                "epsilon_consumed": dp.epsilon,
                # INTENTIONALLY OMITTED: content, doc_id, source_file
            }
            for dp in dp_scores
        ]

Code walkthrough: Differential privacy filter for Tier 2 federated RAG. Key mechanism: the Laplace mechanism adds noise scaled to sensitivity / epsilon; higher epsilon = less noise = better quality but weaker privacy; lower epsilon = more noise = worse quality but stronger privacy. The QUERY_SENSITIVITY_MAP calibrates epsilon by query type: high-sensitivity queries (named individuals) use epsilon=0.1 (large noise); low-sensitivity queries (statistical aggregates) use epsilon=5.0 (small noise). Why it matters: without DP noise, an attacker who queries for “John Smith patient record” and receives a high similarity score learns that John Smith’s record exists in the shard (membership inference attack); with DP noise, the attacker cannot distinguish between “John Smith has a record” and “the noise made a low score look high.” What breaks: the budget_tracker is critical for preventing noise-averaging attacks; if an attacker can make unlimited queries about the same entity, they can average out the Laplace noise and recover the membership signal; the per-query DP budget (epsilon accumulation) prevents this. Takeaway: differential privacy for federated RAG requires BOTH per-query noise AND a global privacy budget tracker; omitting the budget tracker makes the DP guarantee hollow.

from sklearn.metrics.pairwise import cosine_similarity
import numpy as np
from typing import Optional

class ShardRouter:
    """
    For large federations (100+ shards), fan-out-all is too slow.
    ShardRouter predicts which shards are likely relevant
    and queries only those (shard subset routing).
    
    Routing model: each shard has a "shard centroid" embedding
    (average of all document embeddings in that shard).
    Route to shards where shard centroid is closest to query.
    """

    def __init__(
        self,
        shard_centroids: dict[str, list[float]],  # shard_id -> centroid
        top_k_shards: int = 5,  # query this many shards
        min_similarity: float = 0.3,  # exclude very dissimilar shards
    ):
        self.shard_ids = list(shard_centroids.keys())
        self.centroids = np.array(list(shard_centroids.values()))
        self.top_k_shards = top_k_shards
        self.min_similarity = min_similarity

    def route(
        self,
        query_embedding: list[float],
        force_shards: Optional[list[str]] = None,
    ) -> list[str]:
        """
        Route a query to the most relevant shards.
        Returns: list of shard_ids to query.
        
        Args:
            query_embedding: embedded query vector
            force_shards: always include these shards (e.g., compliance requirement)
        """
        query_vec = np.array(query_embedding).reshape(1, -1)
        similarities = cosine_similarity(query_vec, self.centroids)[0]

        # Sort shards by similarity to query
        ranked_shard_indices = np.argsort(similarities)[::-1]
        selected_shards = []

        for idx in ranked_shard_indices:
            if similarities[idx] < self.min_similarity:
                break  # Below minimum relevance threshold
            selected_shards.append(self.shard_ids[idx])
            if len(selected_shards) >= self.top_k_shards:
                break

        # Always include force_shards (regulatory requirement:
        # e.g., always query the compliance shard for certain query types)
        if force_shards:
            for shard_id in force_shards:
                if shard_id not in selected_shards:
                    selected_shards.append(shard_id)

        return selected_shards

    def update_centroid(
        self, shard_id: str, new_centroid: list[float]
    ) -> None:
        """
        Update shard centroid when corpus changes.
        Should be called when: new documents are ingested,
        or the shard's embedding model is updated.
        """
        idx = self.shard_ids.index(shard_id)
        self.centroids[idx] = np.array(new_centroid)

Code walkthrough: Shard router for large federations. Key mechanism: instead of querying all N shards (O(N) latency), the router queries only the top-K shards most similar to the query embedding (O(K) latency where K « N). The “shard centroid” (average of all document embeddings in a shard) is a coarse-grained summary of the shard’s content; querying shards whose centroid is close to the query embedding is likely to return relevant documents. Why it matters: at 100+ shards, fan-out-all produces latency dominated by the slowest shard; with routing, only 5-10 shards are queried, reducing P99 latency by 10-20x. What breaks: shard routing introduces COVERAGE GAPS; if the query is about a topic that is well-represented in a shard with a low centroid similarity (because that shard covers many diverse topics, reducing the centroid’s topic specificity), that shard will not be queried and its documents will be missed; periodically evaluate routing recall (what fraction of relevant shards are included in the routed subset). Takeaway: shard routing trades coverage for latency; calibrate the top_k_shards parameter using routing recall metrics on a held-out test set.

🔁 Flow / Lifecycle

Federated RAG deployment lifecycle:

  1. CONSORTIUM AGREEMENT: Parties agree on: privacy tier, query embedding model (must be standardized), shard API contract, data use agreement.
  2. SHARD DEPLOYMENT: Each organization deploys its own shard service: vector index, retrieval API, privacy filter for the agreed tier.
  3. ORCHESTRATOR DEPLOYMENT: Consortium deploys the orchestrator (hosted by a neutral party or co-hosted).
  4. INTEGRATION TEST: End-to-end test with synthetic queries; verify privacy tier is enforced; verify RRF aggregation is correct.
  5. PRIVACY AUDIT: Privacy auditor reviews a sample of shared results to verify the privacy guarantees are actually provided (not just assumed).
  6. PRODUCTION: Ongoing monitoring: shard latency, privacy budget consumption (Tier 2), SMPC health (Tier 3).
  7. CORPUS UPDATE: When a shard updates its corpus, it notifies the orchestrator to update the shard centroid (if routing is used).

💎 Transferable Wisdom

Reusable Principle: When information must be shared for a collaborative purpose but raw data cannot be shared for privacy reasons, share DERIVED SIGNALS instead of raw data. The quality of collaboration degrades as the signals become more abstract, but privacy improves. The optimal operating point depends on the regulatory environment and the minimum signal quality needed for the collaborative task.

Where Else This Applies:

  1. Fraud detection in banking consortia: Banks cannot share transaction records with each other (banking secrecy, competitive sensitivity). But they can share DERIVED SIGNALS: a hashed entity ID
    • a risk score (not the transactions that generated it). A central aggregator combines risk signals from N banks to detect patterns that no single bank sees. Same privacy tier spectrum: share nothing (useless), share risk score + hashed ID (practical), share full transactions (illegal).
  2. Cross-company supply chain optimization: Companies in the same supply chain cannot share inventory data (competitive sensitivity). They can share AGGREGATED DEMAND SIGNALS (not individual orders) to optimize shared logistics. The signal abstraction level determines the competitive risk.
  3. Privacy-preserving research in genomics: Multiple hospitals cannot share patient genomic data (GDPR, HIPAA). They can share GWAS summary statistics (aggregate associations between variants and traits) without revealing individual genomes. This is the exact Tier 2 pattern: share statistical signals, not raw data.

Industry Applications: Intelligence community (cross- agency information sharing), healthcare consortia (clinical research), financial services (fraud detection), pharmaceutical (pre-competitive research), government (cross-jurisdiction data integration).

💡 The Surprising Truth

The surprising truth about federated RAG is that the hardest part is NOT the technical implementation of the distributed retrieval - it is the GOVERNANCE OF THE DATA SHARING AGREEMENT among consortium members.

The technical components (fan-out queries, RRF aggregation, DP noise) can be built in 2-4 weeks by a competent team. But the CONSORTIUM AGREEMENT - which privacy tier to use, what constitutes a “safe” summary, who hosts the orchestrator, how are breach notifications handled, what happens when a member exits the consortium

  • typically takes 6-18 months of legal negotiation.

The second surprising truth: the privacy guarantee of federated RAG is only as strong as its WEAKEST MEMBER. If one shard operator is compromised, an attacker can modify that shard to return verbatim documents instead of summaries. The federation’s privacy model assumes all shard operators are trustworthy and correctly implement the privacy filter. This is a social contract, not a technical guarantee. SMPC (Tier 3) eliminates this weakness by making the protocol cryptographically enforced, but SMPC’s adoption in production federated RAG is limited by its computational overhead.

The practical implication: invest in the consortium governance framework (data use agreement, breach protocol, audit rights) as much as the technical implementation; federated RAG that is technically correct but has no breach notification protocol is a compliance liability.

🧠 Think About This Before We Continue

Q1: Why does the shard centroid routing strategy (route to shards with highest centroid similarity) fail for shards that cover many diverse topics, and what routing strategy would be more effective?

Hint: The shard centroid is the AVERAGE of all document embeddings; a shard covering diverse topics has a centroid near the center of the embedding space (close to many query types, but rarely the closest to any specific topic). A shard covering only one topic has a centroid close to all queries on that topic but far from all others. The failure mode: diverse-topic shards are often excluded by similarity threshold even when they have highly relevant documents. Better routing strategy: instead of centroid similarity, maintain a per-shard CLUSTER CENTROIDS approach (use K-means on the shard’s document embeddings; maintain K cluster centroids per shard; route if the query is close to ANY cluster centroid). This captures topical breadth without losing specificity.

Q2: How does the choice of query embedding model affect cross-shard retrieval quality in federated RAG, and what happens if different shards use different document embedding models?

Hint: The query embedding and document embedding must be in the same vector space for similarity to be meaningful. If all shards agreed to use the same query encoder (as in the spec), the QUERY is encoded in a common space. But each shard’s DOCUMENTS are embedded using the shard’s own document encoder (which may differ). If Shard A uses BAAI/bge-large-en and Shard B uses sentence-transformers/all-MiniLM-L6-v2, the similarity scores are computed in DIFFERENT SPACES; a score of 0.85 from Shard A is not comparable to 0.85 from Shard B. RRF handles this gracefully (it uses RANKS, not scores), but the underlying retrieval quality depends on each shard’s query-document embedding compatibility. This is a reason to standardize the DOCUMENT encoder across shards (not just the query encoder) as part of the consortium agreement, even though it reduces each shard’s freedom to optimize its embedding model.

Q3 (TYPE G): Implement a minimal federated RAG prototype with two “shards” (two separate in-memory vector stores, each loaded with a different document set). (2) What happens to RRF results when one shard has very few relevant documents (shard with 3 relevant

  • 7 irrelevant) vs another with many relevant documents (shard with 10 relevant)? Does the shard with 10 relevant documents dominate?

Hint: RRF’s k=60 constant dampens the advantage of being ranked #1; a document ranked #1 in one shard gets RRF score 1/61 = 0.0164; a document ranked #3 in two shards gets 1/63 + 1/63 = 0.0318; the multi-shard moderate-rank document BEATS the single-shard top document. This is the key property of RRF for federated RAG: it rewards documents that appear relevant across multiple shards over documents that are strongly relevant in only one shard. For the “10 relevant vs 3 relevant” test: the 10-relevant-document shard contributes more rows to the RRF aggregation, but each row’s contribution is bounded by 1/61 (for rank 1); the total contribution from 10 relevant docs is upper-bounded by sum(1/(60+k) for k=1..10) = 0.14; the 3-relevant-doc shard contributes at most 0.05; the 10-relevant shard DOES have an advantage, but it is proportional to the number of relevant documents, not to raw score magnitude.

🎯 Interview Deep-Dive

Q1: Why can’t you compare raw similarity scores across shards in federated RAG, and what is the correct aggregation approach?

Expected answer: Raw similarity scores are not cross-shard comparable for three reasons: (1) DIFFERENT EMBEDDING MODELS: each shard may use a different embedding model; cosine similarity in Model A’s space means something different from cosine similarity in Model B’s space; a score of 0.85 from Shard A may reflect a weaker match than a score of 0.85 from Shard B if Model A produces more extreme scores. (2) DIFFERENT DOCUMENT DISTRIBUTIONS: a shard with 1M homogeneous documents (all on the same topic) has higher average query-document similarity than a shard with 100K diverse documents; scores from the homogeneous shard are inflated relative to the diverse shard. (3) DIFFERENT SCORING FUNCTIONS: some shards may use cosine similarity, others dot product; these are mathematically different even on the same model. The correct approach: Reciprocal Rank Fusion. Each shard returns its results in RANK ORDER (1, 2, 3…). Ranks are cross-shard comparable by definition: rank 1 means “best in this shard.” RRF formula: sum(1/(k + rank)) for each shard where k=60. Higher RRF score = appears high-ranked in more shards. Documents that are relevant in multiple shards score higher than documents relevant in only one shard.

Q2: What is the membership inference attack in federated RAG and how does differential privacy defend against it?

Expected answer: Membership inference attack: an attacker sends a query that they believe should match a specific document in a specific shard (e.g., “patient John Smith, DOB 1965”). If the shard returns a high similarity score, the attacker infers that John Smith’s records exist in that shard. Without privacy controls, this reveals membership information without accessing the document. Differential privacy defense: before returning similarity scores, the shard adds Laplace noise calibrated to sensitivity/epsilon. With sufficient noise, the attacker cannot distinguish “high score because John Smith’s record exists” from “random noise made a low score look high.” Limitation: DP defends against a SINGLE QUERY attack. If the attacker can send unlimited queries about the same entity and average the responses, the Laplace noise averages to zero and the membership signal is recovered. Defense against noise-averaging: implement a DP BUDGET TRACKER per entity; each query consumes epsilon from the entity’s daily budget; when the budget is exhausted, the shard increases noise or rejects queries about that entity. The budget tracking must be serialized (not concurrent) to prevent race conditions.

Q3: Design the consortium agreement requirements for a federated RAG system across five competing pharmaceutical companies sharing pre-competitive drug discovery data.

Expected answer: Six categories of consortium agreement requirements: (1) DATA CLASSIFICATION: define which documents are eligible for the federation (pre-competitive research vs competitive intelligence); each company labels documents at ingestion; the federation only indexes labeled documents. (2) PRIVACY TIER: consortium must agree on the privacy tier; for pharma pre-competitive data: Tier 1 (masked summaries) with a specific content-masking template (mask: compound names, synthesis routes, trial IDs; retain: general mechanisms, therapeutic area, efficacy signals). (3) ORCHESTRATOR HOSTING: neutral party (industry consortium body, not any member) hosts the orchestrator; audited annually. (4) BREACH PROTOCOL: if a shard returns improperly masked content (verbatim compound names), the orchestrator detects and logs it; the affected shard is suspended within 24 hours; notification to all consortium members within 48 hours. (5) ENTRY AND EXIT: new members can join by deploying a shard and providing test queries for validation; exiting members notify 90 days in advance; the orchestrator removes routing to that shard; the shard decommissions its endpoint. (6) EMBEDDING MODEL GOVERNANCE: changes to the shared query encoder require unanimous consent; a model update requires all shards to re-embed their query-side vectors and a 48-hour validation period before going live.

🌍 Sustainability & Ethics

Environmental Impact:

Federated RAG has a distributed compute footprint: N organizations each run their own vector index and retrieval infrastructure. At N=5 shards with 1M documents each: the total compute footprint is 5x a single centralized RAG system. However: the alternative (centralizing data for RAG) is often not feasible, so federated RAG’s additional compute is justified by the use case. The SMPC overhead (Tier 3) adds 3-10x compute to the aggregation step; for energy-conscious deployments, Tier 1 or Tier 2 is significantly more efficient than Tier 3. At scale (100 shards): the orchestrator becomes a high-compute node; co-locate with renewable energy data centers.

Economic Reality:

Hidden costs of federated RAG: (1) SHARD OPERATIONAL OVERHEAD: each organization must operate its own RAG infrastructure (vector DB, embedding pipeline, retrieval API); the total infrastructure cost is N * single-org RAG cost; (2) CONSORTIUM GOVERNANCE COST: legal negotiation of the data use agreement typically costs $50K-200K in legal fees and 6-18 months of negotiation; this is a one-time cost amortized over the consortium lifetime; (3) INTEGRATION TESTING: verifying that N shards correctly implement the agreed privacy tier requires dedicated testing; privacy audits cost $20K-50K per audit; (4) ORCHESTRATOR HOSTING: the neutral orchestrator must be hosted and maintained; typically hosted by a consortium body at shared cost.

Ethical Implications:

Federated RAG enables knowledge sharing without data centralization, which is generally privacy-positive. However, it creates three ethical risks: (1) FALSE PRIVACY ASSURANCE: organizations may believe federated RAG provides stronger privacy guarantees than it does; Tier 1 (masked summaries) can still leak identifiable information if the masking is inadequate; privacy audits must verify the guarantees, not just trust them; (2) DIFFERENTIAL EXCLUSION: if some consortium members have stronger privacy filtering than others, their documents contribute fewer results to the aggregated answers; this creates an information asymmetry where high-privacy members are effectively penalized for stronger privacy practices; (3) POWER ASYMMETRY: the organization hosting the orchestrator has architectural visibility (query logs, aggregation statistics) that other members do not; this must be governed explicitly in the consortium agreement (orchestrator operator access restrictions, audit rights for all members).

🚒 Incident Runbook

Incident: PII leakage in Tier 1 summaries (FM-01)

  • slo_impact: Correctness - PII leaked to all parties who received the contaminated summaries
  • blast_radius: Every Tier 1 query from this shard since the pipeline was deployed may have leaked PII; 100% of outgoing summaries must be audited; if any PII is confirmed leaked: GDPR breach notification to supervisory authority required within 72 hours

  1. DETECT: Alert: NER PII detector on outgoing summaries identifies PERSON_NAME or DATE entity in a shard’s output. Or: privacy auditor flags specific summaries during routine audit.

  2. CONTAIN: IMMEDIATELY suspend queries to the affected shard (route all queries with “shard N unavailable” message); stop all ongoing Tier 1 summary generation from this shard. Log the exact queries and summaries that triggered the alert for the breach investigation.

  3. DIAGNOSE: (a) Retrieve the source document that generated the summary; (b) verify that the PII filter ran on this document (check filter audit log); (c) if PII filter DID run: identify the PII entity type that was missed (new entity pattern? model failure?); (d) if PII filter did NOT run: identify the code path that skipped the filter (configuration error? exception handling that bypassed the filter?).

  4. REMEDIATE: (a) Update NER model to detect the missed entity type or fix the missed code path; (b) re-run ALL documents in the affected shard through the updated PII filter; (c) rebuild the shard’s summary cache; (d) verify the fix with 100 test documents containing synthetic PII; (e) re-enable the shard after verification.

  5. POSTMORTEM: (a) Conduct GDPR breach assessment: were any data subjects identifiable from the leaked summaries? (b) If yes: notify supervisory authority within 72 hours; notify affected data subjects (Art. 34 if high risk); (c) Add post-summary PII gate as a mandatory check (even if pre-filtering is correct; defense in depth); (d) Add quarterly privacy audits of a random sample of outgoing summaries to detect systematic gaps.

📋 Postmortem Triggers

Trigger 1: A federated RAG consortium’s quarterly privacy audit finds that 3% of outgoing Tier 1 summaries contain organization names that map to identifiable consortium members.

Why postmortem: Organization names in summaries allow the recipient to infer which consortium member contributed the information; this violates the consortium agreement’s requirement for anonymous sourcing. The 3% rate suggests a systematic gap in the PII filter (ORGANIZATION entity type not in the masking list) that has affected all summaries since deployment. Postmortem: (1) audit all summaries in the 90-day look-back period; (2) notify the consortium of the gap; (3) update the PII filter to include ORGANIZATION as a masked entity type; (4) assess whether any trade secret leakage occurred (a competitor learning which member has relevant data is competitive intelligence).

Trigger 2 (silent failure): Answer quality for a specific therapeutic area degrades gradually over 6 months but no alerts fire because no individual query crosses the quality threshold.

Why postmortem: Gradual degradation without threshold breach is the classic silent failure mode: the shard covering that therapeutic area added new document categories over 6 months, making the centroid drift away from its original position; shard routing stopped including it for relevant queries; no individual query’s quality crossed the alert threshold because other shards partially compensated; but the therapeutic area-specific quality degrades below what users expect. Postmortem: implement per-topic quality monitoring (not just aggregate quality), centroid freshness monitoring (alert if centroid has not been updated in 30 days), and routing recall metrics (what fraction of the oracle shards are included in routed subsets).
